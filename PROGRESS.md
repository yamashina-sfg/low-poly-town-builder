# 実装進捗ログ

拡張ロードマップ（フェーズ7〜）の各フェーズ完了時に記録する。

## フェーズ7：ワールドの拡張（チャンク方式への移行）

- 固定10x10グリッドを、歩いた先に自動生成される「チャンク方式」に変更
  （`src/chunkManager.js`, `src/terrain.js`）。1チャンク=10x10タイル
- キャラのいるチャンクが変わるたび周囲3x3チャンクを生成、離れたチャンクは
  `group.visible=false`で描画スキップ（データはメモリ保持）
- 道路の接続判定・セーブ/ロードもグローバルタイル座標でチャンク横断に対応
  （`src/road.js`, `src/save.js`）。自然生成した木は差分保存不要
- 動作確認：東へ長時間歩いて新チャンクが継ぎ目なく生成されること、
  ページ再読み込み後の読込で家・木が同じ場所に再現されることを確認
  （このフェーズはコミット漏れがあったため、フェーズ8と合わせてここに記録）

## フェーズ8：建築バリエーションの拡大

- 建物3種（お店/井戸/倉庫）、家具4種（ベッド/テーブル/椅子/暖炉）、
  装飾物5種（柵/街灯/ベンチ/花壇/看板）を追加。全て既存のInstancedMesh
  プール方式（`src/primitives.js`に共有プール集約）でプロシージャル生成
- 建築メニューをタブ切り替えUI（建物/家具/装飾/地形）に再構成
  （`src/buildMenu.js`, `index.html`）
- `main.js`に`PROCEDURAL_GENERATORS`レジストリを導入し、種類ごとの
  generate関数とseedソルトを一元管理。デバッグパネルの「建物数」は
  住居/お店/井戸/倉庫の合計を表示するよう拡張
- 動作確認：`npm run dev`でビルドメニューの各タブから全12種類を配置し、
  それぞれ見た目・色が正しく生成されることを確認（Playwrightで自動検証）

## フェーズ9：建物の「内部」システム

- 住居タイルをクリックすると、床・壁・天井のみのシンプルな1部屋の室内
  シーンに切り替わる（`src/interior.js`）。カメラのfar(1000)より遠い
  ワールド座標(3000,0,3000)に部屋を常設し、屋外を隠す処理なしで済ませる設計
- 室内の3x3タイルに家具（ベッド/テーブル/椅子/暖炉）をクリックで配置可能。
  レイアウトは住居タイルごとに`userData.indoorFurniture`として保持され、
  再訪しても同じ配置が復元される
- 「外に出る」ボタンとEscキーで元のワールド・元の位置に戻れる。入退室時は
  カメラを即座にスナップさせ、遠距離テレポートによる空白フレームを回避
- 動作確認：家を建てて中に入り、ベッドとテーブルを配置→外に出る→
  再度入って家具が同じ位置に残っていることをPlaywrightで確認

## フェーズ10：時間経過・生活シミュレーション要素

- ゲーム内時間（現実1秒=ゲーム内1分、24分で1日）を`src/gameTime.js`で管理。
  `dayNightCycle.js`をこのゲーム時計に連動させ、朝夕はオレンジ、昼は白っぽく、
  夜は青暗くなるよう色を変化
- 眠気ステータス（`src/playerStatus.js`）を追加。時間経過で徐々に増加し、
  画面左上に時刻と眠気ゲージを表示
- ベッドに近づく（屋内外どちらでも）と「Eキーで眠る」プロンプトが表示され、
  眠ると時間が朝6時まで進み眠気がリセットされる
- 動作確認：ベッド設置→時間経過で時刻・眠気ゲージが進むことを確認→
  ベッドに近づきEキーで睡眠→時刻が06:00、眠気0%に戻ることをPlaywrightで確認。
  Node.jsで深夜/明け方/正午/夕方の光の色も検証

## フェーズ11：経済・リソースシステム

- 木材・お金の所持リソースを`src/economy.js`で管理。建物・家具・装飾ごとに
  コストを設定し、建築時に不足していれば拒否してステータスメッセージを表示
- 木タイルに近づいてEキーで伐採→木材を入手しタイルは更地に戻る。ベッド／木／
  お店のうち最も近いものに応じてEキーの操作内容と画面下部のプロンプトが変化
- お店に近づいてEキーで簡易売買UIを開き、木材⇄お金の交換ができる
- 画面右上に🪵木材・💰お金を常時表示するリソースパネルを追加
- 動作確認：家を建てて資金が正しく減ることを確認→資金不足で2軒目の建築が
  拒否されることを確認→木を伐採して木材獲得→お店で木材売買が正しく資源を
  増減させることをPlaywrightで確認

## フェーズ15：衝突判定の導入

- タイル種別ごとの当たり判定半径を持つ簡易円形衝突判定(`src/collision.js`)を
  追加。建物・木・水・家具・装飾がプレイヤー/NPC/犬の侵入をブロックする。
  グローバルタイル座標から周囲3x3タイルだけを引く方式（O(9)）なので、
  チャンク数が増えても計算コストは増えない
- 「押し出し」方式（重なった分だけ位置を押し戻す）で解決するため、斜め
  移動時に壁沿いへ自然にスライドし、詰まりにくい
- プレイヤー・NPC・犬同士もゆるく押し出し合い、すり抜けないようにした
- 建物・柵・水のあたり判定半径はタイル間隔(2)より2倍の半径合計が大きく
  なるよう調整し、隙間なく並べたときにきちんと壁になるようにした
- 動作確認：単体の建物・木に対する押し出し量が設計通りの境界距離（半径の
  合計）になることを直接検証。建物2つを隙間なく並べた「壁」に向かって
  歩き続けても大きく前進できないこと、単体の木は斜め移動で自然に
  すり抜けられること、プレイヤーとNPCが重なると分離されること
  （境界距離ちょうどまで）、室内で家具を回避しても部屋から出られなく
  ならないことをPlaywrightで確認

## フェーズ16：パフォーマンス基盤の再構築

- 地面タイルをInstancedMesh化（`src/terrain.js`）。タイル1枚ごとの個別
  Geometry/Materialを廃止し、全チャンク共通の1つのInstancedMeshにまとめた。
  レイキャストはinstanceIdからタイルを逆引きする方式に変更。ホバー
  ハイライトは行列に触れずインスタンスカラーだけを更新する
  （`instancing.js`に`setInstanceColor`を追加）
- チャンクの実アンロードを実装（`src/chunkManager.js`）。キャラ周囲3x3
  チャンクだけを読み込み、それより外側は即座にアンロード（地面
  インスタンス・建物/木のInstancedMeshインスタンスを解放）。アンロード前に
  プレイヤーが手を加えたタイルだけを差分キャッシュへ退避し、再訪時に
  同じ内容を復元する（自然生成の木は再生成すれば足りるので保存不要）
- `instancing.js`のInstancedMeshプールに動的拡張を追加。上限(2000)に
  達すると倍の容量の新しいInstancedMeshへ既存インスタンスを複製して
  差し替えるため、上限超過時に新規設置がサイレントに消える不具合を解消
- `getTownStats`の建物数・木の数を、毎回全タイルを舐めるのではなく
  タイル種別が変わった瞬間に増減するインクリメンタルカウンターに変更
  （タイル数はチャンク数からの掛け算のみでO(1)）
- 動作確認：ホバー・クリック建築・道路接続・住居への入室・セーブ/ロードが
  従来通り動作することを確認（実装中に発見した「屋内外切り替え時にホバー
  参照が残る」不具合はその場で修正）。デバッグ用フックで100箇所以上の
  チャンク位置を渦巻き状に訪問してもチャンク数・インスタンス数・
  JSヒープメモリ（約10MB）が完全に一定のまま増加しないことを確認。
  instancing.jsを単体でNode.jsから動かし、2500個のインスタンスを追加
  （上限2000を超過）してもサイレントに失われず、容量が2000→4000へ
  自動拡張され、既存・新規インスタンスの行列が正しく保持されることを確認

## フェーズ19：ゲームループ・目的設計

  `updateLockedButtons`）。万一ロック状態でも選ばれた場合に備え、
  `buildSystem.js`側でも二重に拒否する
- 簡易的な実績システムを追加（`progression.js`のACHIEVEMENTS）：

  ランドマークを発見した（探検家）。達成時に既存のトースト通知
  （`showStatusMessage`）で知らせる。実績・発見数は現状セッション内のみの

- 探索の報酬として、拠点から15タイル以上離れた場所にごく低確率(0.4%)で
  「廃墟」または「特殊な木」（資源が豊富で伐採時の木材が多い）をランドマーク
  として自然生成するようにした（`chunkManager.js`の`getProceduralTileType`）。
  発見（読み込まれたチャンク内に生成された瞬間）は`world.js`の

- 動作確認：Vitestで、評判スコアの計算（装飾数の頭打ち含む）・ランク名の
  しきい値・アンロック判定・実績の新規解除判定・ランドマークが拠点付近には

  スコアの更新が実際に画面に反映されること、セーブ→リロード→読込後も

## フェーズ22：道の拡充とNPCの通勤導線

- 道の種類を増やした：土の道(dirtRoad)・石畳(cobblestone)・橋(bridge)。
  見た目が違っても`ROAD_TYPES`（`road.js`）としてまとめて扱い、互いに
  接続と認識される。橋は水タイルにしか架けられず（`isTilePlaceable`）、
  逆に橋以外の道は水タイルには敷けない。撤去・移動元の橋は水タイルに
  戻る（`world.js`の`getVacatedType`）。フェーズ21の建築プレビュー・
  Rキー回転・移動・撤去にもそのまま対応する（新しい道もrotationYを
  記録できるが、道の見た目自体は接続方向から自動決定されるため実質未使用）
- 建物が道に接続されているかを判定する`isConnectedToRoad`を追加した
  （経済システムと連携するための土台。現時点ではNPCの勤務先割り当て
  条件として使用）
- `populace.js`のNPCに簡易的な目的地システムを追加した：各NPCに
  「家」（住居タイル、`world.js`に新設した`getHouseTiles`から割り当て）
  と「勤務先」（道に接続されたお店のみ、`getShopTiles`＋
  `isConnectedToRoad`でフィルタ）を割り当て、8〜18時は勤務先、それ以外は
  家にいるよう、道タイルを辿る単純な経路（`road.js`の`findRoadPath`、
  幅優先探索）で移動させる。道が見つからない場合は目的地への直線移動に
  フォールバックする
- 実機Playwright検証で、フォールバック時に建物（衝突判定を持つ）の
  中心へ直進させると、衝突による押し戻しで永遠に「到着」を検知できず
  立ち往生する不具合を発見・修正した（`npc.js`の`setDestination`に
  `arriveDistance`引数を追加し、建物へのフォールバック時は衝突半径より
  大きい到達判定距離を使うようにした）
- 道に接続されていないお店は勤務先の割り当て候補から除外することで、
  孤立した店にNPCが向かおうとして詰まる/迷子になることを防いだ
- チャンクアンロード時にベッド/木/お店/水タイルの追跡Setから取り除く
  処理が漏れていた既存の不具合を修正した（`handleTileDispose`。フェーズ22の
  住居/勤務先の割り当てで古いタイル参照を持ち続けないために重要になった）
- 建築メニューのクリック処理を見直した：地形系（更地・木・道・水・橋等）は
  クリックで建築メニューが開き上書きできる、建物・装飾のみクリックで
  移動/撤去メニューが開く、という区別を明確にした（`world.js`の
  `isStructureTile`）。これにより、既存の水タイルをクリックして橋を
  選ぶ、という操作が正しく機能するようになった（実機検証中に発見・修正）
- 動作確認：Vitestで、ROAD_TYPESの接続判定・findRoadPathの経路探索
  （接続時/孤立時/道に隣接していない時）・isTilePlaceableの橋のルール・
  isConnectedToRoad・NPCのsetDestination/arriveDistance・
  getDesiredLocationの時間帯判定を確認（13件追加、計104件）。Playwrightで、
  住居・道・お店を隣接させて実際にNPCが家→（道を辿って）勤務先→
  （時間経過後）家、と移動する様子を目視・座標ログの両方で確認
  （traveling→work→traveling→homeの状態遷移と、継続的に座標が変化する
  ことを確認）。孤立した（道に未接続の）お店には勤務先が割り当てられず、
  全NPCの座標が有限値のまま（詰まる・消えるなどの異常なし）であることを
  確認。橋を水タイルの上に実際に建て、キャラクターがその上まで歩いて
  到達できる（水に阻まれない）ことをスクリーンショットで確認。
  コンソールエラーなし。`npm run lint`・`npm run build`も成功
