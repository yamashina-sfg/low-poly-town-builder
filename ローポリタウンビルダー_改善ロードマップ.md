# ローポリタウンビルダー 改善ロードマップ（フェーズ14〜）

## 現状（フェーズ0〜13完了時点）のレビュー結果

フェーズ0〜13で「歩ける・建てられる・広がる・生活感がある」町作りゲームの
一通りの骨格ができあがった。一方で、実装を横断的にレビューした結果、
「本格的なアプリ」を目指す上では以下の課題が見つかっている。

### 操作性・UX
- 衝突判定が一切なく、建物・木・水・NPCを素通りできてしまう（カメラ以外で
  最大の体感上の問題）
- 建築メニューにコスト表示がなく、クリックして初めて資材不足に気づく
- 操作方法（WASD/E/Esc等）をゲーム内で説明する場所がない
- デバッグ情報（FPS・インスタンス数・チャンク数）とプレイヤー向け機能
  （保存・キャラ色・シード）が同じパネルに混在している
- デスクトップのキーボード・マウス専用（タッチ操作は未対応）

### ゲームとしての面白さ・目的意識
- 目標・進行・スコアが一切ない純サンドボックスで、資材を集めても使い道が
  限定的
- 積み上げた資産がセーブでリセットされる（後述バグ）ため継続する動機が弱い
- 遠方を探索してもまばらな木が生えるだけで、NPCや発見要素は拠点付近に
  固定されており、「広さ」が探索体験として活きていない
- 家の中はどの家も同一（壁紙・広さ共通）

### ビジュアルクオリティ
- 影が一切ない（castShadow/receiveShadow/shadowMap未設定）ため接地感がなく、
  低ポリゲームとしての完成度を大きく下げている
- プレイヤー・NPC共通のキャラモデルに顔や向きを示す特徴がなく、色以外の
  個体差がない
- 室内シーンが1種類のみ

### パフォーマンス／チャンク管理
- 地形タイルが1枚ずつ個別のGeometry/Materialで生成されておりInstancedMesh
  化されていない
- チャンクは可視性を切るだけで一切アンロードされず、長時間探索するほど
  メモリ・処理負荷が増え続ける
- 統計計算(`getTownStats`等)が読み込み済み全タイルを舐めるO(n)処理で、
  0.5秒ごとに実行されるため、探索するほど重くなる
- InstancedMeshプール上限（2000）を超えた際のガードがなく、超過すると
  新規設置が画面に反映されないサイレントな不具合になり得る

### コードの保守性
- main.jsが873行のゴッドファイルで、シーン初期化・入力・建築・経済・時間・
  睡眠・NPC・パーティクル・ミニマップ・カメラの状態がすべて同居している
- 自動テスト・型チェック・静的解析が皆無

### バグ・不具合（実機で確認済み）
- 【重大】室内の家具配置がセーブされない（save.jsがindoorFurnitureを
  一切参照していない）
- 【重大】所持リソース（木材・お金）がセーブされず、ページ再読み込みで
  常に初期値に戻る

これらを踏まえ、フェーズ14以降を「土台の是正」→「基盤の再構築」→
「コンテンツ・体験の強化」の順で進める。

## フェーズ14：データ整合性の是正（バグ修正）

セーブ/ロードの信頼性を最優先で直す。ここが壊れていると、この先どれだけ
コンテンツを積み上げても「積み上がらないゲーム」のままになる。

**Claude Codeプロンプト：**
```
セーブ/ロードのデータ欠落を修正してください。

- src/economy.js の所持リソース（木材・お金）をセーブデータに含め、
  読込時に復元する
- 住居タイルのuserData.indoorFurniture（室内の家具配置）をセーブデータに
  含め、読込時に該当の住居へ復元する
- セーブデータにバージョン番号を持たせ、将来のフォーマット変更に
  耐えられるようにする（現行データの読込に失敗しても壊れないフォールバックを用意）
- 上記の修正が正しく動くことを、家具を置いた家・資材を稼いだ状態で
  保存→ページ再読み込み→読込、のシナリオで確認できる自動テストか
  検証手順を用意する
```

## フェーズ15：衝突判定の導入

建物・木・水・NPCに当たり判定を追加し、「歩く」手触りを根本から改善する。
操作性の観点でカメラの次に優先度が高い項目。

**Claude Codeプロンプト：**
```
キャラクター・NPCの移動に簡易的な衝突判定を追加してください。

- 建物（住居/お店/井戸/倉庫）・木・水タイルには侵入できないようにする
  （タイル単位の当たり判定でよい。厳密なメッシュ衝突は不要）
- 家具・装飾物（ベッド、テーブル、柵など）も同様に通行不可にする
  （ただし室内の家具は部屋が手狭にならない程度の判定サイズに調整する）
- プレイヤーとNPC・NPC同士がすり抜けず、ゆるく押し出し合う程度の
  簡易な反発処理を入れる
- 壁づたいに移動しようとしたときに引っかかって止まらないよう、
  斜め方向にスライドできるようにする（詰まり防止）
- 道路・更地・室内の床は引き続き自由に歩けることを確認する
```

## フェーズ16：パフォーマンス基盤の再構築

コンテンツを増やす前に、チャンクが増え続けても破綻しない土台を作る。

**Claude Codeプロンプト：**
```
チャンク・地形描画のパフォーマンス基盤を見直してください。

- terrain.jsの地面タイルをInstancedMesh化する（現在はタイル1枚ごとに
  個別のPlaneGeometry+Materialを生成しており、チャンクが増えると
  描画コールが線形に増える）。ホバーハイライトの色替えは
  インスタンスカラーの更新で行う
- キャラクターから一定距離以上離れたチャンクは、可視性を切るだけでなく
  実際にGeometry/Materialを破棄してメモリを解放する
  （再訪時は同じシードから再生成すればよい。ただしプレイヤーが手を
  加えたタイルの情報は破棄前にメモリ上のセーブ用データ構造に退避し、
  再生成時に正しく復元する）
- getTownStats・regenerateProceduralTiles・resetTownなど「全タイルを
  毎回舐める」処理を、タイル変更時に差分更新するインクリメンタルな
  カウンター方式に置き換える
- instancing.jsのInstancedMeshプールが上限（2000）に達した場合、
  無音で失敗せずプールを動的に拡張する（新しい大きいInstancedMeshへ
  移行するか、上限を撤廃する）
- 修正後、100チャンク程度探索してもFPSと使用メモリが安定していることを
  確認する
```

## フェーズ17：コード基盤の整備（リファクタリング・テスト導入）

これ以上機能を積み上げる前に、main.jsの肥大化に歯止めをかけ、
自動テストで壊れやすい箇所を守る。

**Claude Codeプロンプト：**
```
main.jsが873行に肥大化し、シーン初期化・入力・建築・経済・時間・睡眠・
NPC・パーティクル・ミニマップ・カメラの状態が1ファイルに同居しています。
責務ごとにモジュール分割してください。

- 例：src/game/ 以下に world.js（チャンク・地形統括）、
  player.js（移動・カメラ・入力）、buildSystem.js（建築・建築メニュー連携）、
  interactions.js（睡眠・伐採・お店などのEキー操作）のように分割する
  （分割方針は既存の各モジュールの責務を尊重し、main.jsは
  「初期化して繋ぐだけ」の薄いエントリーポイントにする）
- グローバルな可変状態（indoorMode、worldSeedなど）は、それを主に
  扱うモジュールに閉じ込め、他モジュールからは関数経由でアクセスさせる
- ESLintとPrettierを導入し、プロジェクトに合った設定でCIから実行できる
  ようにする
- Vitestなど軽量なテストランナーを導入し、economy.js・gameTime.js・
  chunkManager.js・save.jsなど「ロジックが独立していてテストしやすい」
  モジュールから単体テストを書き始める
- 今回発見されたクラス（初期化順序のバグ、符号反転バグ、距離しきい値の
  バグ）を再発防止するテストケースを最低1つずつ含める
```

## フェーズ18：ビジュアル強化

パフォーマンス・保守性の土台が整った上で、見た目の完成度を上げる。

**Claude Codeプロンプト：**
```
低ポリゲームとしてのビジュアル完成度を上げてください。

- DirectionalLightのcastShadowを有効にし、キャラクター・建物・木に
  簡易的な影を落とす（低ポリスタイルを崩さない程度の解像度でよい）
- プレイヤー・NPCのキャラクターモデルに、向きが分かる簡易的な特徴
  （例：帽子のつばを前方に少し伸ばす、体の前面に単色のボタン状の
  ディテールを追加するなど）を加える
- 室内シーンに2〜3種類のバリエーション（壁紙・床材の色違い、部屋の
  広さ違いなど）を追加し、住居ごとにseedから決定論的に選ばれるようにする
- 建物・NPCの配色パレットを見直し、彩度・色相にもう少し幅を持たせて
  単調さを軽減する
```

## フェーズ19：ゲームループ・目的設計

「何のために町を作るのか」を作る、本ロードマップの核心部分。

**Claude Codeプロンプト：**
```
町作りに目的意識を持たせるゲームループを設計・実装してください。

- 「町の評判」または「人口」のような進行指標を導入する。建物の種類数・
  NPCの数・装飾の充実度などから算出し、画面に表示する
- 一定の指標を満たすと解放される新しい建物・装飾（アンロック要素）を
  1〜2種類追加し、プレイに段階的な目標を持たせる
- 簡易的な「実績」システム（初めて家を建てた、NPCが5人になった、
  木材を100集めた等）を追加し、達成時に通知を出す
- 探索の報酬を追加する：拠点から離れた場所に低確率で「廃墟」「資源が
  豊富な特殊な木」などのランドマークを自然生成し、発見する楽しさを作る
```

## フェーズ20：UI/UX仕上げ・オンボーディング

最後に、初めて触る人にも迷わないUIに仕上げる。

**Claude Codeプロンプト：**
```
プレイヤー向けUIを仕上げてください。

- 初回プレイ時に表示される簡単な操作説明（WASD移動、クリックで建築、
  Eキーで操作、Escで外に出る）のオーバーレイを追加し、閉じたら二度と
  出ないようにする
- 建築メニューの各ボタンに必要な木材・お金を表示し、足りない場合は
  ボタンをグレーアウトして選べないようにする
- デバッグパネルを「プレイヤー向け設定パネル」（保存・読込・リセット・
  キャラ色・ミュート）と「デバッグ情報パネル」（FPS・チャンク数・
  インスタンス数など）に分離し、デバッグ情報パネルはデフォルトで
  非表示にして特定のキー操作でのみ開けるようにする
- タッチデバイスでの操作（画面上の仮想スティック・建築メニューのタップ
  操作）に対応する
```

## 進め方の推奨

1. まずフェーズ14（データ整合性）を最優先で直す
   → 資産が消えるバグを放置したまま次に進むと、以降の作業の価値が
   薄まる
2. 次にフェーズ15（衝突判定）
   → カメラ修正と合わせて、操作性の土台をここで完成させる
3. フェーズ16・17（パフォーマンス・コード基盤）は必ずセットで進める
   → コンテンツを増やす前に、増やしても壊れない土台を作る
4. フェーズ18〜20（ビジュアル・ゲームループ・UI）は、土台が固まってから
   着手する。表面的な魅力を先に作ると、後の基盤修正で作り直しが発生しやすい

各フェーズごとに動作確認→GitHub push、を継続する。
